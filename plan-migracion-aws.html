Plan de Migración a la Nube

Comité Paritario — Sistema de Gestión de Reclamos

Curso: Cloud Computing con AWS — CoderHouse

Fecha: Mayo 2026

1. Descripción del Proyecto y Problemática

El Comité Paritario es una aplicación web para la gestión de reclamos y feedback laboral. Los colaboradores envían comentarios con imágenes adjuntas y pueden hacer seguimiento de su solicitud. El área administrativa accede a un panel protegido para gestionar y responder cada reclamo.

CapaTecnología actualUbicación

Base de datosMongoDB (Mongoose)MongoDB Atlas (externo)
Sesionesexpress-session + connect-mongoEn MongoDB Atlas
ImágenesCloudinary (multer-storage-cloudinary)Servicio externo
Archivos estáticosExpress staticDisco local
Secretosdotenv (.env)Archivo local sin cifrado

Problemas identificados:

  • Sin escalado automático: un solo servidor sin tolerancia a fallos.
  • Credenciales expuestas en archivo .env sin cifrado ni auditoría.
  • Dependencia de servicios externos (Atlas, Cloudinary) fuera del ecosistema AWS.
  • Sin monitoreo ni alertas ante errores o saturación.

2. Objetivos de la Migración

  • Centralizar toda la infraestructura en AWS, eliminando dependencias externas.
  • Implementar escalado automático y alta disponibilidad (multi-AZ).
  • Proteger credenciales con AWS Secrets Manager.
  • Reducir latencia de archivos estáticos e imágenes con CloudFront.
  • Implementar observabilidad completa con CloudWatch.

4. Flujo Completo

Usuario enviando un reclamo

  1. El navegador resuelve el dominio vía Route 53 y llega a CloudFront.
  2. CloudFront sirve CSS, JS e imágenes del sitio directamente desde S3 (sin tocar EC2).
  3. El formulario se envía por HTTPS al ALB, que lo enruta a una instancia EC2.
  4. Si hay imágenes adjuntas, multer-s3 las sube al bucket de uploads en S3.
  5. Los datos del reclamo se guardan en DocumentDB mediante Mongoose.
  6. El usuario es redirigido a /gracias.

Administrador gestionando reclamos

  1. El admin accede a /login. La sesión autenticada se almacena en ElastiCache Redis.
  2. En el panel /admin, EC2 consulta DocumentDB para listar los reclamos paginados.
  3. Las imágenes de evidencia se sirven desde CloudFront → S3 con URLs firmadas (acceso privado).
  4. Cada acción queda registrada en CloudWatch Logs.

Inicio de la aplicación (secretos)

  1. Al arrancar, la app consulta Secrets Manager usando el IAM Role de la instancia.
  2. Obtiene MONGO_URI, SESSION_SECRET, etc. en memoria. No existe archivo .env en el servidor.

  • HTTPS obligatorio en ALB con certificado gratuito de ACM.
  • Cifrado en reposo: DocumentDB y S3 con AES-256 (AWS KMS).

8. Plan de Implementación por Fases

Fase 1 — Red, IAM y Seguridad · Semana 1
Crear VPC, subnets públicas y privadas en 2 AZs. Configurar Internet Gateway, NAT Gateway, Route Tables y Security Groups. Crear roles IAM. Cargar secretos en Secrets Manager. Crear bucket S3.
Fase 2 — Base de Datos y Caché · Semana 2
Lanzar cluster DocumentDB en subnet privada. Migrar datos desde MongoDB Atlas con mongodump / mongorestore. Lanzar nodo ElastiCache Redis. Adaptar la app para usar los nuevos endpoints y connect-redis para sesiones.
Fase 3 — Despliegue de la Aplicación · Semana 3
Crear AMI con Amazon Linux 2023 + Node.js 20. Reemplazar multer-storage-cloudinary por multer-s3. Integrar Secrets Manager en el arranque. Lanzar Auto Scaling Group. Crear ALB con HTTPS (ACM).
Fase 4 — CDN, DNS y Estáticos · Semana 4
Subir archivos public/ al bucket S3 estático. Crear distribución CloudFront con origen en S3 y ALB. Configurar dominio en Route 53.
Fase 5 — Observabilidad y Go-Live · Semana 4-5
Configurar CloudWatch Logs en EC2. Crear alarmas y dashboard. Pruebas de carga y funcionales. Corte de DNS al nuevo entorno. Monitoreo activo 48 hs post-migración.

9. Justificación de Decisiones Técnicas

DecisiónJustificación
DocumentDB en lugar de MongoDB AtlasCentraliza la infraestructura en AWS. Mismo VPC, mismo IAM, integración nativa con CloudWatch. Compatible con Mongoose: el código no cambia.
S3 + CloudFront en lugar de CloudinaryLas imágenes son de evidencia, no requieren transformaciones. S3 es más barato y queda dentro del ecosistema AWS.
ElastiCache Redis para sesionesLatencia sub-milisegundo frente a los ~15 ms de MongoDB para sesiones. No compite con el I/O de la base de datos principal.
EC2 + Auto Scaling en lugar de LambdaLa app es stateful: usa sesiones persistentes y renderizado server-side con Handlebars. Lambda no es adecuado para este patrón.
Secrets Manager en lugar de .envCifrado, rotación automática y auditoría vía CloudTrail. Las credenciales nunca se persisten en disco del servidor.
Arquitectura multi-AZGarantiza disponibilidad ante falla de una zona. SLA de EC2 + ALB del 99.99%.

10. Conclusiones

La migración propuesta traslada el sistema Comité Paritario a una arquitectura cloud-native en AWS en 5 fases ejecutables en 5 semanas. Se resuelven los problemas principales: falta de escalabilidad, credenciales expuestas, ausencia de monitoreo y dependencia de proveedores externos.

Los servicios seleccionados son compatibles con el stack Node.js + Mongoose existente, minimizando los cambios de código. El costo estimado de $220.81 USD/mes es reducible a ~$155 USD con Reserved Instances y VPC Endpoints.
